# Чат с шифрованием — как это работает

---

## Архитектура

```
client ←──── TCP 8080 ────→ server
              бинарный протокол

alice.key / bob.key  (RSA ключи, генерируются при регистрации, хранятся только локально)
```

Два компонента:
- `server` — многопоточный, каждый клиент в отдельном потоке (`sys_clone` + `CLONE_VM`), общие данные защищены спинлоком (`xchg`-based)
- `client` — при регистрации генерирует RSA-ключи и сохраняет `username.key`; при входе загружает `.key` и проходит авторизацию; подключается всегда к `127.0.0.1:8080`

---

## Что и где хранится

### Файл `.key` (56 байт, только у пользователя)

```
[0-7]   n   (u64) — модуль
[8-15]  e   (u64) — публичный экспонент (65537)
[16-23] d   (u64) — приватный экспонент
[24-55] username  (32 байта)
```

Приватный ключ `d` никогда не покидает этот файл. Сервер его не получает.

### Сервер (память + `data.bin`)

| Структура | Поля | Размер |
|---|---|---|
| `USER_REC` | `username(32)` + `n(8)` + `e(8)` | 48 байт |
| `CHAT_REC` | `chat_id(8)` + `name(32)` + `chat_n(8)` + `chat_e(8)` | 56 байт |
| `CHAT_KEY_REC` | `chat_id(8)` + `username(32)` + `enc_key(8)` | 48 байт |
| `MSG_REC` | `chat_id(8)` + `enc_sender(8)` + `enc_blocks[43×8]` | 360 байт |

**Сервер знает:** имена пользователей, публичные ключи пользователей `(n, e)`, публичные ключи чатов `(chat_n, chat_e)`, зашифрованные приватные ключи чатов, зашифрованные сообщения.

**Сервер НЕ знает:** приватные ключи пользователей `d`, приватные ключи чатов `chat_d` в открытом виде, содержимое сообщений, имена отправителей.

### Клиент (только в памяти, пока работает)

| | |
|---|---|
| `my_n`, `my_e`, `my_d` | ключевая пара из `.key` файла |
| `lchats[8]` | кеш ключей чатов: `{chat_id, chat_n, chat_d}` |
| `cur_chat_id/n/d` | ключ текущего открытого чата |

## RSA-32

Собственная реализация RSA с 32-битным модулем.

`n = p * q`, где p и q — случайные 16-битные простые числа (≥ 0xC001, через `getrandom`).
Фиксированный публичный экспонент `e = 65537`.

Шифрование и расшифровка — одна функция `rsa_modexp`:
```
encrypt(msg, e, n)  = msg^e mod n
decrypt(enc, d, n)  = enc^d mod n
```

Реализация через square-and-multiply с 128-битным умножением (`mul` + `div` для mulmod).
`n ≈ 2^32` — каждый блок сообщения должен быть меньше `n`.

> **Известное ограничение:** блоки сообщений кодируются по 6 байт (до 2^48),
> что больше `n (~2^32)`. `rsa_modexp` редуцирует `base mod n` перед вычислением,
> поэтому старшие байты блока молча обрезаются → сообщения из не-ASCII или длиннее
> ~4 байт на блок будут повреждены. Требует уменьшения блока до 3 байт.

Сообщение разбивается на 43 блока → до 252 символов на сообщение (формально, до исправления блоков).

**Функции (crypto.inc):**

| Функция | Что делает |
|---|---|
| `rsa_keygen(rdi=&n, rsi=&e, rdx=&d)` | генерирует ключевую пару |
| `rsa_encrypt(rdi=msg, rsi=e, rdx=n)` | `msg^e mod n` |
| `rsa_decrypt(rdi=enc, rsi=d, rdx=n)` | `enc^d mod n` |
| `gen_prime16()` | случайное 16-бит простое (≥ 0xC001) |
| `gen_prime32()` | случайное 32-бит простое (не используется, оставлен) |
| `mod_inverse(rdi=a, rsi=m)` | `a⁻¹ mod m` (расширенный алгоритм Евклида) |

## Протокол

Бинарный, поверх TCP. Каждый пакет начинается с 1 байта — кода команды или ответа.

### Команды клиент → сервер

| Код | Команда | Данные |
|---|---|---|
| 0x01 | `CMD_REGISTER` | `username(32)` + `n(8)` + `e(8)` |
| 0x02 | `CMD_AUTH_START` | `username(32)` |
| 0x03 | `CMD_AUTH_RESP` | `challenge(8)` |
| 0x04 | `CMD_CHAT_AUTH` | `chat_id(8)` |
| 0x05 | `CMD_CHAT_AUTH_RESP` | `chat_id(8)` + `challenge(8)` |
| 0x06 | `CMD_LIST_CHATS` | — |
| 0x07 | `CMD_CREATE_CHAT` | `name(32)` + `chat_n(8)` + `chat_e(8)` |
| 0x08 | `CMD_GET_KEY` | `chat_id(8)` |
| 0x09 | `CMD_SEND_KEY` | `chat_id(8)` + `username(32)` + `enc_key(8)` |
| 0x0A | `CMD_GET_PUBKEY` | `username(32)` |
| 0x0B | `CMD_SEND_MSG` | `chat_id(8)` + `enc_sender(8)` + `43×enc_block(8)` |
| 0x0C | `CMD_GET_MSGS` | `chat_id(8)` |
| 0x0D | `CMD_LIST_MY_CHATS` | — (только чаты текущего пользователя) |

### Ответы сервер → клиент

| Код | Ответ | Данные |
|---|---|---|
| 0x10 | `RESP_OK` | — |
| 0x11 | `RESP_ERR` | — |
| 0x12 | `RESP_CHALLENGE` | `enc_challenge(8)` |
| 0x13 | `RESP_CHAT_LIST` | `count(8)` + N × `CHAT_REC(56)` |
| 0x14 | `RESP_CHAT_KEY` | `enc_key(8)` + `chat_n(8)` |
| 0x15 | `RESP_MSGS` | `count(8)` + N × `MSG_REC(360)` |
| 0x16 | `RESP_PUBKEY` | `n(8)` + `e(8)` |
| 0x17 | `RESP_CHAT_ID` | `chat_id(8)` |

## Основные процессы

### Регистрация

```
Клиент                              Сервер
→ CMD_REGISTER + username + n + e
                                    Сохраняет USER_REC(username, n, e)
                                    Записывает data.bin
                                    ← RESP_OK
```

### Вход (challenge-response)

```
Клиент                              Сервер
→ CMD_AUTH_START + username
                                    challenge = getrandom() mod user_n  ← challenge < n
                                    enc = rsa_encrypt(challenge, user_e, user_n)
                                    ← RESP_CHALLENGE + enc
rsa_decrypt(enc, my_d, my_n) → challenge
→ CMD_AUTH_RESP + challenge
                                    Сравнивает с оригиналом
                                    ← RESP_OK
```

Доказывает владение приватным ключом без его передачи по сети.
Challenge берётся по модулю `n` чтобы гарантировать `challenge < n` (необходимо для корректного RSA).

### Создание чата

```
Клиент (Alice)                      Сервер
rsa_keygen() → chat_n, chat_e, chat_d   ← генерируется ЛОКАЛЬНО
→ CMD_CREATE_CHAT + name + chat_n + chat_e
                                    Сохраняет CHAT_REC (chat_d не получает)
                                    ← RESP_CHAT_ID + chat_id
Сохраняет {chat_id, chat_n, chat_d} локально

enc_key = rsa_encrypt(chat_d, my_e, my_n)   ← шифруем для себя
→ CMD_SEND_KEY + chat_id + username + enc_key
                                    Сохраняет CHAT_KEY_REC
                                    ← RESP_OK
```

`chat_d` рождается у клиента и в открытом виде никуда не уходит.

### Приглашение пользователя

```
Клиент (Alice)                      Сервер
→ CMD_GET_PUBKEY + "bob"
                                    ← RESP_PUBKEY + bob_n + bob_e
Берёт chat_d из локального кеша
enc_key = rsa_encrypt(chat_d, bob_e, bob_n)   ← шифруем для Bob
→ CMD_SEND_KEY + chat_id + "bob" + enc_key
                                    Сохраняет CHAT_KEY_REC для Bob
                                    ← RESP_OK
```

### Вход в чат

```
Клиент (Bob)                        Сервер
→ CMD_GET_KEY + chat_id
                                    Ищет CHAT_KEY_REC для (chat_id, bob)
                                    ← RESP_CHAT_KEY + enc_key + chat_n
chat_d = rsa_decrypt(enc_key, my_d, my_n)
Сохраняет {chat_id, chat_n, chat_d} локально

→ CMD_CHAT_AUTH + chat_id           ← аутентификация чата
                                    enc = rsa_encrypt(challenge, chat_e, chat_n)
                                    ← RESP_CHALLENGE + enc
rsa_decrypt(enc, chat_d, chat_n) → challenge   ← доказывает знание chat_d
→ CMD_CHAT_AUTH_RESP + chat_id + challenge
                                    ← RESP_OK
```

### Отправка сообщения

```
Клиент                              Сервер
enc_sender = rsa_encrypt(username[0..5], chat_e, chat_n)
для каждого 6-байтного блока текста:
  enc_block[i] = rsa_encrypt(block[i], chat_e, chat_n)
→ CMD_SEND_MSG + chat_id + enc_sender + 43×enc_block
                                    Сохраняет MSG_REC
                                    ← RESP_OK
```

### Чтение сообщений

```
Клиент                              Сервер
→ CMD_GET_MSGS + chat_id
                                    ← RESP_MSGS + count
                                    ← N × MSG_REC
Для каждого:
  sender_name = rsa_decrypt(enc_sender, chat_d, chat_n)
  для каждого блока:
    block[i] = rsa_decrypt(enc_block[i], chat_d, chat_n)
  выводим: "sender_name: сообщение"
```

## Персистентность (data.bin)

При каждом изменении (регистрация, создание чата, приглашение, сообщение) сервер перезаписывает `data.bin`.

**Формат:**
```
[0-7]   "CHATDAT1"  ← magic bytes для проверки формата
[8+]    user_count(8) + chat_count(8) + chat_key_count(8) +
        msg_count(8) + next_chat_id(8) +
        users[16×48] + chats[16×56] +
        chat_keys[128×48] + messages[256×360]
```

Итого: **~100 КБ**. При старте сервер читает magic bytes, и если они совпадают — загружает весь блок одним `sys_read`.

Сбросить состояние: `make cleandata` — удаляет `*.key` и `data.bin`.

## UI и навигация

Клиент использует ANSI escape `\033[2J\033[H` для очистки терминала при каждом переходе между экранами.

**Экран входа:**
```
1. Войти  2. Регистрация:
Имя пользователя:
```
Пустое имя — клиент завершается. IP не запрашивается, всегда `127.0.0.1:8080`.

**Главное меню:**
```
=== МЕНЮ ===
1. Создать чат
2. Войти в чат
3. Пригласить в чат
4. Выйти
```

**Экран "Войти в чат"** — показывает список чатов, в которых есть доступ у текущего пользователя (`CMD_LIST_MY_CHATS`). Enter без ввода — возврат в меню.

**Экран "Пригласить в чат"** — аналогично показывает только доступные чаты. Enter без ввода — возврат в меню.

**Экран чата** — автоматически загружает и показывает все сообщения в формате `имя: текст`. Снизу поле ввода (до 252 символов). Enter без ввода — выход в меню.

**Ошибки** — выводится конкретное сообщение (`Ошибка: ...`) с паузой `[ Enter для продолжения ]` перед возвратом в меню.

## Известные особенности

**RSA base reduction** — при отправке сообщения клиент шифрует данные публичным ключом чата. Если значение `>= chat_n`, деление переполняется (SIGFPE). Исправлено в `rsa_modexp`: base редуцируется по модулю n перед вычислением (`base = base mod n`).

**mod_inverse overflow** — в расширенном алгоритме Евклида (`crypto.inc:135`) промежуточное произведение `q * s` переполняло 64-битный регистр при 32-битных простых (φ(n) ≈ 2^63, q * s → 2^109). Исправлено переходом на 16-битные простые (`gen_prime16`): φ(n) < 2^32, максимальное произведение ≈ 2^47.

**challenge > n** — сервер генерировал 64-битный challenge очищая только бит 63, что при n ≈ 2^32 давало challenge >> n почти всегда. После RSA round-trip клиент возвращал `challenge mod n ≠ challenge` → авторизация всегда падала. Исправлено: challenge берётся как `getrandom() mod n` (`server.asm`, оба места генерации challenge).

**Блоки сообщений > n** — блоки кодируются по 6 байт (до 2^48), что превышает n ≈ 2^32. `rsa_modexp` молча обрезает старшие байты через `base mod n`. Сообщения из ASCII-символов в нижних 3 байтах работают, остальные — повреждаются. Не исправлено: требует уменьшения блока до 3 байт и пересчёта `MAX_MSG_BLOCKS`.

**SO_REUSEADDR** — сервер устанавливает `SO_REUSEADDR` перед `bind()`, что позволяет немедленно перезапускаться без ожидания освобождения порта от TIME_WAIT соединений.
